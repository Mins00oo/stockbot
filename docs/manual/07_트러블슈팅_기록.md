## 1. `python`이 "SRE module mismatch"로 죽음
- **상황**: 페어링 키 생성하려고 cmd에서 `python -c "import secrets; print(secrets.token_urlsafe(32))"` 실행.
- **오류**: `AssertionError: SRE module mismatch` (`sre_compile.py`), `D:\devtools\Python310\Lib\...`를 로드 중.
- **원인**: 사용자 환경변수 `PYTHONHOME = D:\devtools\Python310`(3.10)이 설정돼 있어, 실행된 python(3.12)이 **3.10의 표준 라이브러리를 강제 로드** → 인터프리터(3.12) vs Lib(3.10) 버전 불일치.
- **시도**: `where python`(3.12·3.10·3.14·WindowsApps 여러 개 확인), `PYTHONHOME`/`PYTHONPATH` 점검 → User scope `PYTHONHOME=3.10` 발견.
- **해결**: `[Environment]::SetEnvironmentVariable("PYTHONHOME", $null, "User")`로 삭제 → **새 터미널** → `python --version` = 3.12.10 정상. (PYTHONHOME은 보통 설정하면 안 되는 값.)

## 2. 프론트 `npm install` 의존성 충돌(ERESOLVE → ETARGET)
- **상황**: 프론트 첫 `npm install`.
- **오류**: `ERESOLVE`(expo-constants `~19.0.0` ↔ expo-router는 `^56.0.18` 요구), `--legacy-peer-deps` 시 `ETARGET No matching version found for expo-constants@~19.0.0`(존재하지 않는 버전).
- **원인**: 빌드 에이전트가 `package.json`의 **expo-* 버전을 추정값(placeholder)** 으로 넣음. SDK56은 expo-* 패키지가 **통일 버전(56.x)** 인데 옛 개별 버전(19 등)으로 적혀 있었고 일부는 아예 존재하지 않음.
- **시도**: `--legacy-peer-deps`(ETARGET로 실패) → 원인이 "없는 버전"임을 확인.
- **해결**: `package.json`을 **최소 유효 상태로 축소**(expo + 비-expo 라이브러리만) → `npm install` → **`npx expo install <expo/RN 패키지들>`** 로 SDK56 정답 버전 자동 채움. 이후 `react-dom↔react` peer 충돌은 **`.npmrc`에 `legacy-peer-deps=true`** 추가로 영구 해결(expo 표준 방식). → 이제 어느 PC든 `npm install` 한 번.

## 3. `docker compose up` — PG18 볼륨 마운트 에러
- **상황**: 백엔드 Postgres를 `docker compose up`으로 기동.
- **오류**: `in 18+, these Docker images ... there appears to be PostgreSQL data in /var/lib/postgresql/data (unused mount/volume)`.
- **원인**: **PostgreSQL 18+ 도커 이미지는 볼륨을 `/var/lib/postgresql`에 마운트**해야 함(내부에 버전별 하위폴더 저장). docker-compose가 17 이하 방식 `/var/lib/postgresql/data`로 돼 있었음.
- **시도**: 에러 메시지 + `docker-compose.yml` 확인.
- **해결**: 마운트를 `stockbot_pgdata:/var/lib/postgresql`로 수정 → `docker compose down -v`(잘못 만들어진 볼륨 제거, 실데이터 없어 안전) → `docker compose up -d`.
- **메모**: `/var/lib/postgresql`은 **컨테이너 내부(리눅스) 경로**고 named volume은 도커가 관리 → 윈도우 폴더로 안 보임(정상).

## 4. `npx expo-doctor` 2건 실패 (app.json 스키마 + typescript)
- **상황**: 빌드 전 선제 점검으로 `npx expo-doctor`.
- **오류**: `app.json` → `should NOT have additional property 'newArchEnabled'`, `'splash'`; 패키지 → `typescript expected ~6.0.3, found 5.9.3`.
- **원인**: SDK56에서 **`newArchEnabled` 필드 제거**(새 아키텍처 기본 on), **`splash`는 expo-splash-screen 플러그인으로 이동**. typescript는 SDK56 기대 버전과 불일치.
- **시도**: `app.json` 구조 확인.
- **해결**: `newArchEnabled` 제거 + `splash`를 `expo-splash-screen` 플러그인 설정으로 이전 + `npx expo install --fix`로 typescript `~6.0.3` 정렬 → **expo-doctor 18/18 통과**.

## 5. iOS `eas build` pod install 실패 — worklets 누락
- **상황**: `npx eas build --profile development --platform ios` (클라우드 pod install 단계).
- **오류**: `[Reanimated] react-native-worklets package isn't installed. Please install a version between 0.8.x ... to use Reanimated 4.3.1` → `Invalid RNReanimated.podspec` → `pod install exited with non-zero code: 1`.
- **원인**: **Reanimated 4.x가 worklets 런타임을 별도 패키지 `react-native-worklets`(~0.8.x)로 분리**했는데 미설치. (`expo install` 시 reanimated만 넣고 companion 누락.)
- **시도**: 에러 분석으로 worklets 의존성 식별.
- **해결**: `npx expo install react-native-worklets` 추가.

## 6. iOS `eas build` — Apple 라이선스 동의서 에러
- **상황**: `eas build ... ios` 중 Apple 인증/크레덴셜 단계.
- **오류**: `App Store Connect has agreement updates that must be resolved` (+ EU DSA trader status 요구) → `build command failed`.
- **원인**: **Apple Developer Program 라이선스 동의서 업데이트 미수락** + 디지털서비스법(DSA) trader status 미입력. (우리 코드 문제 아님 — Apple 계정 행정.)
- **시도**: 에러가 Apple 계정 이슈임을 식별.
- **해결**: https://developer.apple.com/account 에서 **Account Holder가 업데이트된 동의서 수락** (+ App Store Connect에서 trader status 입력). 코드 수정 없음.

## 7. `uv run alembic upgrade head` — cp949 UnicodeDecodeError
- **상황**: DB 기동 후 마이그레이션 적용 `uv run alembic upgrade head`.
- **오류**: `UnicodeDecodeError: 'cp949' codec can't decode byte 0xe2 in position 125` (`alembic.ini` 읽기).
- **원인**: `alembic.ini` 주석에 **em-dash `—`(UTF-8 비ASCII)** 가 있었음. Alembic(최신)이 ini를 **`encoding="locale"`** 로 읽는데, **한국어 Windows의 locale = cp949** 라 그 UTF-8 바이트(0xe2)를 못 디코드.
- **시도**: `alembic.ini` 확인 → 6번 줄 주석의 em-dash가 position 125와 일치함을 확인.
- **해결**: 해당 주석 2줄 삭제(파일을 순수 ASCII로). → 설정 파일은 ASCII로 두는 게 안전.

## 8. `uv run alembic upgrade head` — ProactorEventLoop ↔ psycopg async (Windows)
- **상황**: alembic.ini 고친 뒤 다시 `uv run alembic upgrade head`.
- **오류**: `psycopg.InterfaceError: Psycopg cannot use the 'ProactorEventLoop' to run in async mode. Please use a compatible event loop ... SelectorEventLoop ...`.
- **원인**: **Windows 기본 asyncio 이벤트 루프 = ProactorEventLoop**, 그런데 **psycopg 비동기 모드는 SelectorEventLoop만 지원.** alembic `env.py`가 *불필요하게* async 엔진(`asyncio.run`)으로 작성돼 있어 이 충돌을 떠안음. (리눅스는 무관 — Windows 전용.)
- **시도**: `env.py`·`main.py` 분석. (마이그레이션은 async가 필요 없다는 점 확인.)
- **해결**:
  - `alembic/env.py` → **동기(sync) 엔진으로 재작성** (마이그레이션은 순차 작업 → async 불필요 → 이벤트 루프 문제 자체 제거).
  - `main.py` → 서버는 의도적으로 async라, **Windows일 때만 `asyncio.set_event_loop_policy(WindowsSelectorEventLoopPolicy())`** 강제(`sys.platform=="win32"` 가드 → 리눅스 무해).

## 9. 앱 버튼이 흰 배경에 흰 글자 + 위치 깨짐 (NativeWind 간섭)
- **상황**: dev build로 앱 첫 실행 → 온보딩 인트로의 "연결 시작하기" 버튼.
- **오류**: 파란 버튼이 안 보이고(흰 배경에 흰 글자), 가운데정렬·높이도 사라져 위치가 깨짐.
- **원인**: 앱은 스타일을 **전부 인라인 `style`(`theme/tokens.ts`)** 로 그리는데 **NativeWind v4가 babel/metro에 켜져** 모든 컴포넌트를 감쌈. `PrimaryButton`의 `Pressable`이 style을 **함수형**(`({pressed}) => [...]`)으로 넘기는데 NativeWind가 이를 제대로 안 흘려보내 배경·높이·정렬이 드롭됨. (자식 `<Text>`는 일반 객체 style이라 `color:#fff`만 적용 → 흰 글자만.) `className` 사용은 **0건** — NativeWind는 안 쓰는데 간섭만 함.
- **시도**: `className` 전체 검색(0건) → "미사용 + 간섭원" 확인.
- **해결**: **NativeWind 완전 제거** — babel(`jsxImportSource`/`nativewind/babel`), metro(`withNativeWind`), `_layout`의 `global.css` import, 파일 삭제(`global.css`·`tailwind.config.js`·`nativewind-env.d.ts`), `tsconfig` include, `package.json`(nativewind·tailwindcss·prettier-plugin-tailwindcss), `.prettierrc` plugins. 스타일은 인라인+`tokens.ts`로 통일. `npx expo start --clear` 후 리로드(JS 전용이라 네이티브 재빌드 불필요).
- **교훈**: 안 쓰는 스타일/변환 시스템을 빌드에 켜두지 말 것.

## 10. 앱 "네트워크 연결을 확인해주세요" (실기기 localhost)
- **상황**: 실기기(폰) 앱에서 페어링/연결 시 백엔드 호출.
- **오류**: "네트워크 연결을 확인해주세요" — 요청이 백엔드에 닿지 못함.
- **원인**: `EXPO_PUBLIC_API_URL=http://localhost:8000`인데 **실기기에서 `localhost`는 폰 자기 자신** → PC의 백엔드를 못 찾음. (게다가 기본 uvicorn은 `127.0.0.1`만 바인딩.)
- **시도**: `client.ts` baseURL 확인 → localhost 기본값 식별.
- **해결**: (1) `.env`의 `EXPO_PUBLIC_API_URL`을 **PC LAN IP**(`ipconfig`→IPv4, 예 `http://192.168.0.10:8000`), (2) 백엔드 **`--host 0.0.0.0`** 실행, (3) 방화벽 8000 인바운드 + 폰·PC 같은 와이파이, (4) `EXPO_PUBLIC_*`는 번들 시점 값이라 `.env` 변경 후 `npx expo start --clear`. (자세히 `05`·`01`)
- **교훈**: 실기기 개발에선 localhost 금지 — LAN IP + `0.0.0.0` 바인딩이 기본.

## 공통 교훈
- **스캐폴딩(자동 생성) 산출물은 최신 SDK/플랫폼 관례와 어긋날 수 있음** → 첫 구동 전 `expo-doctor`(프론트)·`alembic`(백엔드) 등으로 선제 점검.
- **설정 파일은 ASCII로** (Windows 로케일 디코드 이슈 회피).

- **버전은 도구가 정하게**: 프론트는 `expo install`, 의존성 잠금은 lockfile + `.npmrc`.
