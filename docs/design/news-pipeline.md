- **문서 버전**: v1
- **작성일**: 2026-06-24
- **상태**: 설계(기획). 구현 전.
- **범위**: 뉴스/공시 **수집 → 정규화 → 중복제거 → 종목매칭 → 처리구조(`헤드라인→본문내용→결론→LLM의 생각`)** 까지의 큰 플로우.

## 0. 큰 그림 (End-to-End 플로우)

```
 [소스들]
 네이버API · 증권RSS · Finnhub · GDELT · AlphaVantage · DART · SEC EDGAR
        │  (각 소스 어댑터가 폴링)
        ▼
 ① 정규화        → 소스마다 다른 응답을 공통 RawItem(제목·원문링크·요약snippet·발행시각·출처·소스태그)로
        ▼
 ② Exact 중복제거 → canonical URL(고유키) + 정규화 텍스트 해시. 동일/재게재 즉시 컷 (무ML, 거의 공짜)
        ▼
 ③ 종목 매칭     → 소스태그 → 별칭사전 → (애매한 것만)Haiku. 기사→종목(신뢰도·근거문구)
        ▼
 ④ 관련성 필터   → 내 관심·보유 종목에 안 걸리면 폐기 (Tier0 무료필터)
        ▼
 ⑤ Near-dup     → SimHash(char 3-gram). 리워딩된 같은 기사 병합 (종목·최근기간 내)
        ▼
 ⑥ 이벤트 클러스터 → 임베딩 + 종목블로킹 + 시간윈도우. 여러 매체의 "같은 사건"을 1 스토리로,
                    대표기사 선정 + 출처수(corroboration) 집계
        ▼
 ⑦ 1차 분류(Haiku) → 스토리별 관련도·감정·신규성·한줄요약·escalate 판정
        ▼
 ⑧ (escalate 시) 깊은 분석(Opus) → 본문 일시 취득 → 4부 구조 생성, 안전장치 적용
        ▼
 ⑨ 저장(Memory) / 알림(Discord, 급한 것) / 리포트(앱)
```

**핵심 순서 규칙(조사 근거):** *Exact → Near-dup → Event-cluster* 순서를 지킨다. 클러스터링을 먼저 하면 와이어 재게재가 클러스터 크기를 부풀려 "출처 수" 신호가 거짓이 된다. **출처 수는 항상 *서로 다른 매체* 기준으로 센다.**

**모델 개입 지점 (비용의 핵심):** 중복 제거는 **비싼 추론 LLM에 넘기기 *전*** 에 끝난다.
- ②Exact·⑤Near-dup = **순수 코드**(모델 0).
- ⑥이벤트클러스터 = **임베딩(작은 벡터 모델)** — 추론 LLM 아닌 의미 유사도 계산.
- ③종목매칭 = 사전(공짜) + 애매한 잔차만 작은 Haiku.
- **비싼 추론 LLM(⑦Haiku 분류 · ⑧Opus 분석)은 dedup·필터가 모두 끝난 뒤에만** 호출 → 같은 기사에 LLM 비용을 중복 지불하지 않는다.

## 2. ① 정규화 — 공통 `RawItem`

모든 소스를 하나의 모양으로 통일:
```
RawItem {
  source            // naver | rss:hankyung | finnhub | dart | edgar | gdelt | av
  source_type       // news | filing | macro
  title             // 헤드라인 (HTML 태그/엔티티 제거 후)
  original_url      // 발행사 원문 링크 (네이버 mirror 아님 — 본문 취득·attribution용)
  snippet           // 소스가 준 짧은 요약 (네이버 description 등, 본문 아님)
  published_at      // 발행 시각 (정렬·인과 타임스탬프 검증용)
  source_tickers[]  // 소스가 준 종목 태그 (Finnhub/DART/EDGAR), 없으면 빈 값
  raw_ref           // 원본 식별자(oid/aid, accessionNumber 등)
}
```
- **HTML 클렌징**: 네이버/RSS의 `<b>`, `&quot;`/`&amp;`/`&#39;` 등 엔티티·태그 제거.
- **KR 보일러플레이트 제거**(해시/유사도 정확도용): `[속보][단독][종합]` 태그, `(서울=연합뉴스)` 데이트라인, `○○○ 기자 / 매체명 / 무단전재 금지` 꼬리말.

## 3. ② Exact 중복제거 (무ML, 제일 먼저, 거의 공짜)

**두 개의 키만으로 대부분의 중복 제거 (전체 중복의 30~50%):**

**(A) Canonical URL** — DB 고유 인덱스로 재유입 자동 차단:
1. scheme/host 소문자, https 강제, 기본포트·트레일링 슬래시·`#fragment` 제거
2. 추적 파라미터 제거: `utm_*`, `fbclid`, `gclid`, `ref`, `spm`, `from` 등. **네이버는 `oid`/`aid`는 유지(기사 식별), `sid`/`rc`/`cds`는 제거.** (블랙리스트보다 *의미있는 것만 화이트리스트* 가 안전)
3. AMP/모바일 정규화: `/amp`, `?outputType=amp` 제거, `m.` → `www.`
4. 단축링크·리다이렉트(30x, Google News 래퍼)는 최종 URL까지 따라간 뒤 정규화
5. 가능하면 페이지의 `<link rel=canonical>`/`og:url` 우선 — 단 일부 발행사가 홈/페이월로 잘못 지정하므로 *맹신 말고* 규칙 기반을 폴백으로

**(B) 정규화 텍스트 해시** — 다른 URL로 재게재된 같은 와이어 기사(연합/로이터 신디케이션) 컷:
- 제목(+가능 시 snippet) NFKC 정규화 → 보일러플레이트 제거 → 공백 정리 → **SHA-256** → `exact_hash`(인덱스). 동일 해시 = 동일 기사. O(1), 언어 무관.

→ 이 두 키(`canonical_url` + `exact_hash`)가 exact 레이어의 전부. **이걸 건너뛰고 ML부터 만들지 말 것**(흔한 과설계 실수 — 정규화 안 하면 출처수·빈도 신호가 쓰레기가 됨).

## 4. ③ 종목 매칭 + ④ 관련성 필터

**레이어드(싼 것부터), 모든 자동 태깅엔 *근거 문구(verbatim span)* 첨부 → 사람이 감사 가능:**
1. **소스 태그** (최고 정밀): Finnhub/DART/EDGAR가 준 종목/CIK/corp_code 그대로 신뢰.
2. **별칭 사전** (주력 무료 필터): `종목마스터` 기반 종목별 별칭 — 정식명·통칭·브랜드/제품명·티커·CEO명. 단어경계 매칭. **종목코드(005930) 검색은 네이버/RSS에서 안 됨.**
   - ⚠️ 모호성 방어: 흔한 단어/계열사 충돌('삼성'→전자·SDI·물산…)은 스톱리스트 + "주변에 회사 맥락어 필요" 규칙.
3. **Haiku 잔차** (애매/미등록 별칭만): watchlist를 주고 거기로만 매핑(우주 전체 아님), 티커+신뢰도+근거 span 반환. 환각 방지 위해 *원문 인용 span 필수*.
4. **피드백 학습**: Haiku가 새 별칭 발견 → 사전에 추가 → 다음부턴 공짜 매칭. (정정 루프와 동일 메커니즘, 코드/지식파일 경로)

**④ 관련성 필터(Tier0)**: 내 관심·보유 종목에 하나도 안 걸리면 폐기. *비싼 임베딩·LLM을 무관 기사에 절대 쓰지 않음.*

## 5. ⑤ Near-dup + ⑥ 이벤트 클러스터 (단계적 도입)

> 우리 볼륨은 **연 수만 건으로 아주 작다** → 화려한 인덱스보다 **단순+브루트포스**가 정답. 아래는 *필요해질 때* 단계적으로.

### ⑤ Near-dup (리워딩된 같은 기사)
- **64-bit SimHash over 문자 3-gram.** Hamming distance ≤ 3이면 병합.
- **왜 char-gram?** 한국어는 조사/어미가 붙어("삼성전자가/는/를") 단어 단위 shingling이 같은 문장을 다르게 봄 → 재현율 추락. **문자 n-gram은 형태소 분석기(KoNLPy) 없이** 조사 변형에 강하고 KR+EN 혼용도 처리. (1인 봇 실용 기본값)
- 볼륨이 작아 새 기사 fingerprint를 **최근 2~3일치와 브루트포스 비교** — LSH 인덱스 불필요. 정말 커지면 `datasketch` MinHashLSH(Jaccard≥0.8).

### ⑥ 이벤트 클러스터 ("한 사건, N개 매체")
- 제목+리드를 **다국어 임베딩**(`paraphrase-multilingual-MiniLM-L12-v2`, 한국어 학습됨, 토크나이저 작업 불필요)으로 인코딩.
- **종목 블로킹 + 시간윈도우(24~48h)** 로 비교 대상 축소 → 같은 종목/같은 기간만 클러스터(교차종목 오병합 방지).
- **단일 패스 증분 클러스터링**: 새 기사가 기존 클러스터 중심과 cosine ≥ ~0.72면 합류, 아니면 새 클러스터. 라이브러리 불필요, 스트리밍 친화. (배치 재클러스터 품질이 필요하면 그때 agglomerative.)
- **대표기사 선정**: 최초 발행(선발주자, 주식봇에 유리) 또는 권위 소스(연합/로이터/IR). 나머지는 멤버로 보존, `출처수`(서로 다른 매체) 집계 → *출처수↑ = 사건 신뢰/중요도 신호*.

> ⚠️ **임계값은 그대로 못 믿는다.** cosine은 ~0.80~0.85에 몰려 있어 빌린 숫자(0.85/0.72)는 오탐 양산 → **내 KR/EN 기사쌍 소량을 직접 라벨링해 캘리브레이션** 필수. SimHash k, MinHash Jaccard도 동일.
> ⚠️ 오병합은 *판단의 문제*(같은 날 등급하향+목표가컷은 한 사건? 두 사건?). 비대표 기사를 **하드 삭제 말고 보존**해 되돌릴 수 있게.

### 빌드 (3단계 모두 범위에 포함 — 사용자 확정)
세 단계 모두 **설계·범위에 포함**한다. 아래는 *구현 순서*일 뿐 스코프 컷이 아니다.
- **P1**: ② Exact 중복제거. (대부분의 중복 제거, 비용 ~0)
- **P2**: ⑤ SimHash near-dup. (리워딩 카피 병합)
- **P3**: ⑥ 임베딩+이벤트 클러스터. ("한 스토리 N출처" 집계)
- 점진 롤아웃은 가능하나(P1부터 가치 발생), 최종적으로 3단계 모두 구현한다.

## 6. ⑦⑧ 처리 구조 — `헤드라인 → 본문내용 → 결론 → LLM의 생각`

네가 원한 4부 구조 = **스토리(클러스터) 1건의 정식 표현.** 단 ⚠️ **저작권 제약**으로 *본문은 verbatim 저장 금지* → 아래 정책으로 구현.

### 4부 구조 정의
| 파트 | 내용 | 생성 | 저장? |
|---|---|---|---|
| **헤드라인** | 발행사 제목(짧아서 verbatim OK) 또는 살짝 패러프레이즈 | 그대로 | ✅ 저장 |
| **본문내용** | **LLM이 우리 말로 요약**(transformative). 원문 문장/구조 베끼지 않음, 골격형 압축+우리 프레이밍, 원문 대체 못 할 길이로 | LLM | ✅ 요약만 저장 |
| **결론** | 핵심 한 줄 takeaway | LLM | ✅ 저장 |
| **LLM의 생각** | 안전장치 붙은 해석/투자판단(가설·신뢰도·근거). 명백히 우리 창작 → 위험 최저 | Opus | ✅ 저장 |

### 처리 깊이 (확정: A — 본문 요약과 깊은 생각을 분리)
모든 기사에 Opus를 돌리면 비싸다 → **본문 요약(싸다)** 과 **깊은 생각(Opus, 비싸다)** 을 분리:
- **관련 스토리 *전부*** → 헤드라인 + **본문내용**(본문 *일시 취득* → **Haiku 요약**) + **결론**. *본문은 항상 읽고 요약됨* (사용자 기대 충족).
- **escalate된 중요 스토리 + 앱/채팅 요청 시** → 추가로 **LLM의 생각**(Opus, 안전장치 4종 분석).

→ 모든 관련 스토리가 4부 *틀*을 갖고 **본문도 항상 존재**하며, **Opus의 깊은 생각만 중요도에 비례.** 관련 종목 필터로 양이 줄어(하루 수십 건) 본문 fetch+Haiku 요약은 부담 없음.

### 본문 취득 (Full일 때만, 일시적)
- 발행사 `original_url` 페이지를 가져와 **readability 추출**(폴백 체인: `trafilatura` → `readability-lxml` → `newspaper4k`). 단일 라이브러리로 모든 깨진 페이지 못 잡음.
- **예의 수집**: 정직한 User-Agent, 낮은 동시성, throttle, 429/403 시 지수 백오프 + `Retry-After`, **robots.txt 준수**. 페이월/등록월 페이지는 *스킵*하고 snippet만 사용.
- JS 렌더링 페이지(headless 필요)는 무겁고 차단 위험 → 가급적 회피.
- **취득한 본문 전체는 절대 영속 저장 안 함** → LLM에 먹여 본문내용/결론 생성 후 폐기(필요 시 짧은 TTL 캐시만, 시스템 오브 레코드 아님).

### 안전장치 4종 (LLM의 생각에 강제 — product-spec §6-4)
① 시장/섹터 베타 잔차 먼저(잔차≈0 → "그냥 시장 따라감") ② 뉴스 타임스탬프가 가격 움직임보다 *먼저*인지 검증 ③ "원인 불명" 당당히 허용 ④ 수치·인명은 원문에 있는 것만(없으면 인용 거부). **모든 인과는 사실 아닌 *가설* 로 표기 + 신뢰도 + 근거(헤드라인·시각·출처).**

## 7. 저작권/데이터 정책 (요약 — 상세 근거는 research)

**KR 저작권법 §30 사적이용 + US fair use** 안에 머무는 설계:
- **영속 저장**: 헤드라인 / `original_url`·매체·발행시각 / **본문내용=transformative 요약** / 결론 / LLM의 생각 / (선택)snippet.
- **취득 후 폐기**: 추출한 *본문 전체* (DB/디스크 기록 안 함).
- **전제**: 시스템은 **완전 사적**(나 1인, 공개 엔드포인트·공유·재배포 없음). 요약을 외부로 내보내는 순간 면책 붕괴.
- ⚠️ "요약"이라는 단어가 면죄부 아님 — 원문 문장/구조를 그대로 따라가는 substitutive 요약은 침해 가능. *진짜 transformative + 사적 + 비재배포* 라서 안전.
- ⚠️ 네이버/RSS 자체 ToS(쿼터·재배포 금지)도 별도 준수. 법률 자문 아님 — 사적·비상업 1인 봇 기준 위험 저감 패턴.
