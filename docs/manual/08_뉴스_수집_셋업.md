# 08. 뉴스 수집(1단계) 셋업 — 외부 키 발급 + 실행·검증

- **작성일**: 2026-06-27
- **대상**: 뉴스 파이프라인 **1단계**(수집 → 정규화 → 중복제거 → 저장). LLM 분석은 다음 단계.
- **무엇을 하나**: 백엔드를 실행하면 스케줄러가 **토스 보유종목**의 뉴스를 주기적으로 긁어와
  중복을 제거하고 DB에 저장한다. (US=Finnhub, KR=네이버 검색 API)

---

## 1. 발급할 외부 키

| 키 | 시장 | 발급처 | 필수? |
|---|---|---|---|
| `FINNHUB_API_KEY` | 🇺🇸 US 뉴스 | https://finnhub.io (무료 가입 → API Key) | US 종목 보유 시 |
| `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET` | 🇰🇷 KR 뉴스 | https://developers.naver.com | KR 종목 보유 시 |

> 둘 다 **없어도 서버는 정상 기동**한다. 키가 없는 소스는 그냥 건너뛴다(graceful skip).
> 보유종목이 KR뿐이면 네이버 키만, US뿐이면 Finnhub 키만 있으면 된다.

### 1-1. 네이버 검색 API 키 발급 (KR)
1. https://developers.naver.com 로그인 → **Application → 애플리케이션 등록**.
2. 사용 API에서 **검색**을 선택.
3. 환경 추가는 **WEB 설정**으로 두고(서버에서 호출), 등록 완료.
4. 발급된 **Client ID / Client Secret**을 복사.
5. 무료 한도: 25,000콜/일. (보유 수십 종목 × 주기 폴링이면 넉넉)

### 1-2. Finnhub 키 (US)
- 이미 종목 상세 화면용으로 `FINNHUB_API_KEY`를 세팅했다면 **그대로 재사용**된다(뉴스도 같은 키).
- 없으면 finnhub.io 무료 가입 후 키 발급. 무료 한도 ~60콜/분.

## 2. 세팅 위치 — `backend/.env`

`backend/.env`에 아래를 추가(`.env.example`에도 동일 주석 있음):

```dotenv
# US 뉴스 + 펀더멘털
FINNHUB_API_KEY=발급받은_finnhub_키

# KR 뉴스 (네이버 검색 API)
NAVER_CLIENT_ID=발급받은_client_id
NAVER_CLIENT_SECRET=발급받은_client_secret

# 스케줄러 (선택 — 기본값으로 충분)
SCHEDULER_ENABLED=true          # false로 끄면 자동 수집 중지(수동 트리거는 가능)
NEWS_POLL_INTERVAL_MINUTES=15   # 수집 주기(분)
NEWS_LOOKBACK_DAYS=2            # 소스 조회 시 며칠치(중복은 자동 제거)
```

## 4. 스케줄러는 백엔드와 같이 돈다

- 백엔드(`uvicorn`)를 실행하면 **같은 프로세스 안에서 스케줄러가 자동 기동**한다.
  (FastAPI lifespan, tech-stack "옵션 A". 추후 별도 워커로 분리 가능하게 독립 모듈로 작성.)
- 기동 ~15초 후 1차 수집이 한 번 돌고, 이후 `NEWS_POLL_INTERVAL_MINUTES`마다 반복.
- 끄려면 `.env`에 `SCHEDULER_ENABLED=false`.

```bash
cd backend
uv run uvicorn stockbot.main:app --host 0.0.0.0 --port 8000
# 로그에 "Scheduler started: news ingest every 15 min" 가 보이면 정상
```

## 5. 검증 (e2e)

> 전제: 토스 계좌가 연결돼 있어야 한다(보유종목을 가져와야 하므로). 미연결이면 수집은
> `skipped: not_connected`로 그냥 건너뛴다(에러 아님).

### 5-1. 즉시 1회 수집 (스케줄 안 기다리고)
```bash
curl -X POST http://localhost:8000/news/ingest/run \
  -H "X-Pairing-Key: <페어링키>"
# → {"fetched":N,"stored":M,"deduped":K,"symbols":S,"skipped":null}
```
- `fetched`=소스가 준 기사 수, `stored`=새로 저장, `deduped`=중복으로 버림.

### 5-2. 저장된 뉴스 조회
```bash
curl "http://localhost:8000/news?limit=20" -H "X-Pairing-Key: <페어링키>"
curl "http://localhost:8000/news?symbol=AAPL" -H "X-Pairing-Key: <페어링키>"
```

### 5-3. 중복제거 동작 확인
- `POST /news/ingest/run`을 **연속 2번** 호출 → 두 번째는 `stored`≈0, `deduped`↑ 이면
  중복제거(canonical_url + exact_hash)가 정상 동작하는 것.

## 6. 트러블슈팅

| 증상 | 원인/조치 |
|---|---|
| `stored:0, fetched:0, skipped:not_connected` | 토스 미연결 → 온보딩에서 계좌 연결 먼저 |
| KR 뉴스가 안 들어옴 | `NAVER_CLIENT_ID/SECRET` 누락/오타. 서버 로그 `Naver ... status 401` 확인 |
| US 뉴스가 안 들어옴 | `FINNHUB_API_KEY` 누락 또는 한도초과(429). 로그 `Finnhub company-news ... status` 확인 |
| KR 회사명 매칭 노이즈 | 1단계는 회사명 쿼리 기반이라 동명이의가 섞일 수 있음. 별칭사전·스톱리스트는 다음 단계 |

## 7. 1단계 범위 / 다음 단계

- ✅ **이번(1단계)**: 수집 → 정규화(RawItem) → ②Exact 중복제거(canonical_url + exact_hash) → 저장.
- ⏭ **다음**: ⑤near-dup(SimHash) · ⑥이벤트클러스터(임베딩) · ⑦Haiku 1차분류 · ⑧Opus 깊은분석(4부구조)
  · 별칭사전 · 공시(DART/EDGAR) · 디스코드 알림 · 리포트. (`docs/design/news-pipeline.md` 참고)
